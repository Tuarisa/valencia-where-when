# Progress: pipeline

## Original Goal

pipeline — должен быть пайплайн, сбор из многих разных источников с возможностью расширения, частота обновления из каждого должна настраиваться отдельно, базируясь на том как часто где-либо появляются обновления, есть каналы в телеграме где апдейты каждые 10 минут, его можно читать часто, а есть где новые посты не каждый день. Сайты так же вроде обновляются реже. Запуск должен быть параллельный. Все полученные данные это сырые объекты. Дальше должна быть обработка, которая их дедуплицирует и энричит и приводит в порядок, дополняет, размечает тегами, и все такое. Это должно быть ясно и четко объяснено мне. Если необходимо поменять структуру БД или кода чтобы соответствовать спецификации - так тому и быть.

## Learnings

- `ingestAll()` (ingest.ts:241-254) fetches ALL enabled sources SEQUENTIALLY (for-loop) every run; `ingestSource` has a single global 6h freshness floor (minIntervalHours=6) — NOT per-source, NOT adaptive.
- `sources` table (schema.sql:5-16) has only `last_fetched`; no interval/next_due_at/etag/last_modified. Cadence columns must be added (additive).
- Parser selection is a hardcoded if/else branch (ingest.ts:214-225), unlike the normalizer registry. Add a parallel PARSER_REGISTRY to make "add a source" data+registration only.
- `NORMALIZER_REGISTRY` (normalize.ts:16) + idempotent pending-select (l.53-58) already exist — the extension seam is half-built; only `api:hemisferic` registered today.
- Raw append-only + raw-identity dedup ALREADY satisfied: `upsertSourceItem` + `sourceItemHash` (util.ts:33-48), UNIQUE dedup_hash. "Сырые объекты" requirement met.
- Vercel reality (confirmed across vercel-workflow + constitution): Hobby=1 cron/day; Workflow SDK has NO scheduler. Per-source 10-min cadence needs an EXTERNAL tick (GH Actions `*/10`, reusing 001-D1) → a DB-driven dispatcher that selects only `next_due_at <= now()` sources.
- Adaptive-polling canon (cited): `interval ≈ observed_gap × 0.33`, clamp [min,max] (telegram 10m–3h, web 1h–24h); conditional GET (ETag/If-Modified-Since→304); back-off on unchanged, speed-up on change; backoff+jitter on failure.
- GOTCHA: Telegram `t.me/s/` previews + scraped HTML often DON'T honor ETag/Last-Modified and change every request. Reliable change signal = "did this fetch produce NEW source_items?" (item-hash delta). Use HTTP conditional GET only where honored.
- `Promise.allSettled` is the per-source failure-isolation primitive; `ingestSource` already try/catches + logs error source_runs, so it's isolation-ready — only needs a bounded parallel pool (concurrency 4-6, no new dep needed).
- Stage order reconciliation: constitution V says ingest→normalize→dedup→score→tag→geo→enrich, but dedup+card-enrich specs refine to dedup-after-normalize + enrich-after-dedup + enrich-before-geo (enrich feeds clean venue to geo). Umbrella adopts: ingest→normalize→dedup→enrich→tag→score→geo. Fixes 001 T025 (dedup wrongly placed "between tag and geo").
- dedup/enrich are NOT wired into run.ts yet (run.ts = ingest→normalize→score→tag→geo only); enrich.ts doesn't exist → low-friction to insert + reorder.
